Status: active constitutional directive
Date: 2026-07-10
Owner: 윤

## 0. Core Thesis

GPAO-T is not an external device attached to OpenClaw.

GPAO-T is the project of absorbing GPAO, BEAI Package, Context Mesh, Knowledge Loop, T-cell theory, authority governance, replay, and self-growth into OpenClaw's local runtime body until OpenClaw is transformed into a new personal AI operating system.

The correct metaphor is not:

```text
OpenClaw + plugin + external GPAO service
```

The correct metaphor is:

```text
Buy a Ferrari, understand Ferrari engineering deeply, then rebuild it into a different supercar.
```

This means:

- OpenClaw is the current body/material.
- Pure OpenClaw source is the engineering anatomy object.
- GPAO / BEAI Package / T-cell / Context Mesh are the replacement and enhancement technologies.
- Codex is the development and synthesis tool.
- nbeai GitHub repositories are the primary source library.
- Hidden previous-work result storage is deprecated and must not be used as the source of truth.

## 1. Source Authority Order

Use sources in this order:

1. User's current instruction and live correction.
2. Current live OpenClaw state on the user's PC.
3. Pure OpenClaw GitHub source anatomy under the local lab.
4. nbeai GitHub repositories and their local clones.
5. T-cell canonical theory documents in the workspace.
6. Local Context Mesh hits after turn-start retrieval.
7. Prior local docs only when they are still present, linked, and not contradicted.

Do not use hidden previous-work recording repositories as active truth. They may be treated as discarded history unless the user explicitly reopens them.

## 3. Absorption Rule

Prefer internal absorption over external connection.

Allowed end-state direction:

- replace OpenClaw modules one by one
- modify OpenClaw modules in place when justified
- wrap temporarily only as a migration step
- delete OpenClaw code when its function has been safely replaced
- preserve OpenClaw strengths where they reduce risk
- improve OpenClaw weaknesses with GPAO/T-cell/BEAI engineering

Rejected default direction:

- keep OpenClaw unchanged and attach GPAO as a side service
- rely on plugin-only integration as the final architecture
- build a disconnected GPAO-T surface beside OpenClaw
- treat OpenClaw improvement as the product goal
- treat hidden prior-work records as stronger than current source evidence

## 4. Engineering Standard

GPAO-T must be built with serious engineering ambition, not only problem-solving patches.

The project must study:

- OpenClaw internals
- other AI OS / local agent runtime projects
- model APIs and provider behavior
- local gateway / daemon / service lifecycle
- prompt and context engineering
- tool approval and sandbox models
- model routing and fallback
- UI/UX for multi-chat work surfaces
- memory, context, replay, and self-growth systems
- security, credentials, and authority boundaries

The goal is not merely "make it work." The goal is to become technically capable enough to build a better operating system around rented models.

## 5. OpenClaw Source Treatment

OpenClaw source may eventually be:

- analyzed
- forked
- copied into lab branches
- modified
- replaced
- deleted
- reorganized

But every such action must follow:

```text
source anatomy
  -> live comparison
  -> replacement rationale
  -> lab patch
  -> tests / visual QA / replay
  -> rollback plan
  -> explicit live approval when live OpenClaw is touched
```

No broad live mutation is allowed merely because the final product will replace OpenClaw.

## 6. Ferrari Rule

The Ferrari rule:

> If we are rebuilding a Ferrari into a different supercar, we must understand both Ferrari and automotive engineering.

Applied to GPAO-T:

- Understand OpenClaw deeply before changing it.
- Understand model APIs deeply before wrapping them.
- Understand local runtime engineering before making daemon/gateway decisions.
- Understand UI/UX work rhythm before redesigning the dashboard.
- Understand T-cell theory as runtime engineering, not decoration.
- Understand other AI operating systems and agent frameworks as engineering references, not competitors to copy blindly.

## 8. Authority Boundary

Allowed without additional approval:

- inspect nbeai GitHub repositories
- inspect local clones
- inspect pure OpenClaw source
- update local planning/design/engineering documents
- create lab/fork patches in workspace-controlled directories
- run local non-live tests

Requires explicit approval:

- mutate live OpenClaw
- restart live Gateway
- run live Telegram/OpenClaw model turns
- change credentials/tokens
- activate external connectors or automation
- promote durable memory
- push GitHub
- publish/deploy
- delete unknown user work

## 9. Operating Reminder

Every future GPAO-T development pass should ask:

```text
Does this move OpenClaw closer to becoming GPAO-T from the inside?
```

If the answer is no, the work is probably a side experiment, not the main line.
