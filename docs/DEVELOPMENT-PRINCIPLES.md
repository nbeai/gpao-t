# GPAO-T Development Principles

Status: operating guidance for product development

GPAO-T is not a rejection of OpenClaw or Codex. It should absorb proven foundations and recombine them around the GPAO-T semantic and growth kernel.

## Product Standard

- Stability should aim for OpenClaw-grade local runtime reliability.
- Work comfort, flexibility, and UX should aim for Codex-grade flow.
- GPAO-T originality should come from T-cell, Context Mesh, Memory Wiki, and Self-Growth Loop changing user experience in concrete runtime behavior.

## Build Judgment

Reuse or closely mirror what OpenClaw already solves well:

- local runtime
- gateway
- doctor / health
- event log
- update / rollback
- sandbox and elevated execution distinction
- model fallback

Actively learn from what Codex already proves well:

- thread / workspace flow
- composer
- command palette
- approval UX
- progress / status
- developer-friendly work rhythm

Use BEAI Harness `design.md` before building GPAO-T visual surfaces. GPAO-T UI must feel like an operating desk: current work, context, evidence, growth, authority, and next safe action must be visible before decoration.

Build as GPAO-T-owned kernel work:

- active-target recovery
- Context Mesh admission
- Memory Wiki candidate supply
- T-cell runtime decision units
- authority and governance semantics
- connector permission boundaries
- install / update / rollback readiness
- Local Control Center design contract
- self-growth proposal and application gates
- user-visible continuity and recovery
- research-grounded skill ecosystem registry
- skill routing, skill quality gates, and skill growth signals

## Implementation Rule

Every new GPAO-T concept must map down to:

- function
- state
- replay
- audit
- UI state
- user-visible behavior

If a concept cannot be inspected through runtime output, tests, replay evidence, audit evidence, or Control Center state, it is not yet implemented.

If a UI concept cannot be mapped to `docs/LOCAL-CONTROL-CENTER-DESIGN-RECIPE.md`, it is not yet ready for visual implementation.

If a skill concept cannot be mapped to `docs/04-skill-ecosystem/GPAO-T-SKILL-ECOSYSTEM-MASTER-PLAN-ko.md`, it is not yet ready for runtime attachment.

## Runtime Rule

T-cell is not decorative theory. It must affect runtime decisions:

- how the current request is interpreted
- which memory or candidate evidence is admitted
- which actions are blocked
- which model/tool path is selected
- which self-growth proposal is created

Context Mesh and Memory Wiki are not raw storage. They are evidence and candidate supply layers for T-cell admission.

Self-Growth Loop may automate candidate discovery, but durable memory promotion, OS rule mutation, connector activation, token storage, external send, and external action require replay, audit, rollback, and approval gates. Application gates may record readiness, but live mutation must remain blocked until a separate reversible apply engine exists.

Install, update, rollback, daemon activation, deployment, external download, and secret storage must start as visible readiness contracts. Real executors require explicit approval, recovery evidence, and post-operation verification.

Fast user experience matters. GPAO-T should keep fast path and deep path separate so ordinary turns stay light and deep reasoning happens only when needed.

## Skill Ecosystem Rule

GPAO-T skills are not role prompts. They are research-grounded operating units.

Every skill pack must start from a real user problem, study strong references and practical domain standards, extract the T-cell operating principle, then define execution procedure, output artifacts, quality gates, replay cases, authority boundaries, and growth signals.

The default skill policy is active automation with narrow authority gates:

- skill discovery, routing, draft generation, quality inspection, replay draft, and upgrade proposal are allowed
- durable memory promotion, live skill mutation, OS rule mutation, external send, deployment, public release, legal/financial/security/destructive actions, and paid connector use require approval and replay/audit/rollback gates

The first skill ecosystem implementation surface is:

```text
gpao-t skill ecosystem
gpao-t skill manifest
gpao-t skill packs [category]
gpao-t skill inspect <skill-pack-id>
gpao-t skill route <text>
gpao-t skill readiness
```
