# GPAO-T Agent Runtime to Personal Growth OS Matrix v0.1

Date: 2026-07-13
Status: active comparison matrix

## Position

OpenClaw is a separate product and comparison target. GPAO-T should not be
measured as a patch over OpenClaw or as a feature-count clone. The comparison
asks whether GPAO-T rises from agent runtime to Personal Growth OS.

The short criterion:

> OpenClaw can run an agent. GPAO-T must understand, admit, govern, verify, and
> grow with the user.

## Matrix

| Axis | Agent Runtime Baseline | GPAO-T Personal Growth OS Evidence |
| --- | --- | --- |
| Product identity | runtime/dashboard product | `nBeAI. GPAO-T` user-visible identity; OpenClaw only as attribution, compatibility, history, or comparison |
| Runtime body | gateway, dashboard, workspace | `~/.gpao-t`, `gpao-t`, `ai.nbeai.gpao-t`, port `18799`, receipts, install/rollback/snapshot |
| Intent recovery | chat input routed to agent | T-cell admission, active target recovery, task packet, current-goal recovery |
| Context handling | session recall or memory search | Context Mesh candidates separated from admitted anchors and support-only material |
| Memory | stored/retrieved facts | raw source -> candidate -> replay -> local apply -> post-apply replay -> keep/rollback |
| Authority | method/tool approval | action authority classes: memory, connector, external send, live mutation, durable promotion |
| Execution | gateway/tool dispatch | local runtime, CLI/MCP, model/tool router, evidence receipts |
| Self-evaluation | answer/result review | preflight, post-answer replay, doctor, verify, closeout, false-pass guard |
| Growth | manual prompt or note updates | self-growth candidate loop with review-only default and explicit promotion gate |
| Recovery | restart/session recovery | snapshot, rollback, reinstall, route crawl, cleanup evidence |
| UX feel | chat surface | workspace OS: current work, progress, memory, authority, session, growth, next safe action |
| Release truth | package exists | source seal, identity audit, Docker smoke contract, local blocker classification |

## Evidence Metrics

- Product identity leakage count
- Runtime ascent coverage
- Context admission precision
- Authority separation failures
- Growth proof strength
- Surface legibility across desktop/mobile
- OpenClaw baseline respect
- Release truth: package, install, health, rollback, Docker status
- First progress signal latency
- Conversation QA pass rate
- Context recovery latency
- Admitted/support/rejected candidate counts

## P0 Comparison Rules

1. Compare OpenClaw only from a pure source/lab reference, not a modified
   GPAO-T lab tree.
2. Treat OpenClaw as another product, not GPAO-T's visible identity or active
   product substrate.
3. Do not frame the output as "we have more features." Frame it as an operating
   layer ascent: runtime -> context/authority/growth OS.
4. Any remaining user-visible OpenClaw/Gateway/ClawHub residue is a product
   identity leak unless it is in explicit attribution, comparison,
   compatibility, or historical evidence.
5. Release/test-team wording must say what is verified, what is blocked by
   environment, and what remains a planned lane.

## Current Phase 2 Reading

GPAO-T already has the first real product evidence for a Personal Growth OS:

- independent product identity and runtime boundary
- local install, health, rollback, reinstall, and receipts
- Context Mesh and Knowledge Loop as OS organs rather than simple memory notes
- T-cell admission/replay as a candidate judgment mechanism
- user-facing work/session surface moving away from raw chat

The remaining Phase 2 burden is not to prove that the idea is plausible. It is
to seal the source/runtime/package evidence so a tester can distinguish:

- verified product behavior
- environment blocker
- comparison-only OpenClaw reference
- historical absorption evidence
- future enhancement lane
