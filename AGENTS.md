# BEAI Harness Project Rules

1. Do not modify files before a patch-plan exists.
2. Do not claim completion before verify evidence exists.
3. AI/developer must run scenario-first verification before asking the user for final testing.
4. If verification fails, enter recover mode before broad changes.
5. Keep v0 scope and non-goals intact unless the user explicitly approves a scope change.
6. External send, credential handling, memory write, cron creation, deployment, and destructive changes require explicit approval.
7. Prefer draft/proposal output before mutation.
8. Keep user-facing progress notes updated when the user is beginner or owner mode.
9. If a command is blocked by phase gate, explain the missing evidence and recommend the next safe action.
10. Ask the user only for what AI cannot do: intent judgment, preferences, real-world approval, and final lived-context feedback.
